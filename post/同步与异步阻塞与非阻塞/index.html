同步与异步、阻塞与非阻塞的区别

对不这些概念的理解，网上查询了下，不同的人有不同的理解，有些差别很大。 这里主要是给出我自己的理解，如果与大家不同，还请指教，不胜感激。

同步与异步

同步与异步关注的是消息通信机制， - 所谓同步，就是在发出一个“调用”时，在没有得到结果之前，该“调用”就不返回。但是一旦调用返回，就得到返回值了。就是由调用者主动等待这个“调用”的结果。 - 所谓异步，就是“调用”在发出之后，这个调用就直接返回了，所以没有返回结果。调用者不会立刻得到结果。而是在“调用”发出后，“被调用者”通过状态、通知来通知调用者，或者通过回调函数处理这个调用。

阻塞与非阻塞

阻塞与非阻塞关注的是程序在等待调用结果时的状态。 - 所谓阻塞，就是指调用结果返回之前，当前线程会被挂起。调用线程只会在得到结果之后才会返回。 - 所谓非阻塞就是指在不能立即得到结果之前，该调用不会阻塞当前线程。

unix网络编程中将IO模型分为5类：阻塞IO，非阻塞IO，IO复用，信号驱动，异步IO。

  1. 阻塞IO就是哪种recv，read，一直等，等到有结果才返回。
  2. 非阻塞IO就是立即返回，设置描述符为非阻塞，但是要进程自己一直检查是否可读。
  3. IO多路复用（select，poll，epoll）其实也是阻塞的，不过可以用来等很多描述符，比起阻塞有了进步，可以算有点异步了，但需要阻塞着检查是否可读。对同一个描述符的IO操作也是有序的。
  4. 信号驱动采用信号机制等待，有了更多的进步，不用监视描述符了，而且不用阻塞着等待数据到来，被动等待信号通知，由信号处理程序处理。但对同一个描述符的IO操作还是有序的。
  5. 异步IO，发送IO请求后，不需要等待，也不再需要发送IO请求获取结果了，等到通知后，其实是系统帮你把数据读取好了的，你等到的通知也不再是要求你去读写IO了，而是告诉你IO请求过程已经结束了。你要做的就是可以处理数据了。且同一个描述符上可能同时存在很多请求。

线程池

一般在不同的系统下，可以创建的最大的线程数是有限制的，而创建的线程过多也不太好，太费内存了，这时我们可以使用线程池。 系统启动一个新线程的成本是比较高的，因为它涉及与操作系统的交互。在这种情形下，使用线程池可以很好地提升性能，尤其是当程序中需要创建大量生存期很短暂的线程时，更应该考虑使用线程池。

多线程之间通信

多线程编程难处在于由于多线程共享进程的内存空间，对于内存的修改，需要同步，不然可能会产生错误的结果，而线程同步需要加锁，不恰当的编程可能导致死锁现象。多线程之间可以使用队列或者管道来通信，能更好地解决编码难的问题。

注意：由于python的GIL锁的原因，python中的多线程不是真正的多线程，在同一时刻只能有一个线程运行，对于计算密集型的多线程显然不适用，但是对于IO密集型的操作(例如爬虫),由于cpu都在等待该IO操作，所以单cpu等待该操作也不会有啥影响，此时多线程编码能极大提高速度，当然我们还可以使用协程来进一步提高性能。

import selectors
import socket

#创建selector类，具体选择select、pool还是epool看操作系统平台
sele = selectors.DefaultSelector()

#回调函数
def accept(sock,mask):
    conn,addr = sock.accept()
    print('接受',conn,'从',addr)
    #设置为非阻塞
    conn.setblocking(False)
    #注册回调函数，当socket句柄为可读取时，返回句柄
    sele.register(conn,selectors.EVENT_READ,read)


def read(conn,mask):
    data = conn.recv(1024)
    if data:
        print('打印',repr(data),'给',conn)
        conn.send(data)
    else:
        print('正在关闭',conn)
        sele.unregister(conn)
        conn.close()
sock = socket.socket()
sock.bind(('localhost',8080))
sock.listen(100)
sock.setblocking(False)
#注册可读事件，当socket句柄可读取时返回句柄，并不是调用回掉函数
sele.register(sock,selectors.EVENT_READ,accept)


#事件循环，不断轮询事件状态
while True:
    events = sele.select()
    for key,mask in events:
        callback = key.data
        callback(key.fileobj,mask)

协程

协程可不是那个“携程在手，说走就走的那个啊，喂”，上述IO复用和回调的写法太蛋疼了，为了解决这个问题。提出了协程，协程就是子线程。轻量级线程。 它可以解决很多问题： 1. 回调模式编程复杂度高 2. 同步编程的并发性不高 3. 多线程编程需要线程间同步，需要使用lock，而且线程的切换需要开销。

协程的优点： 1. 采用同步的方式编写异步代码 2. 使用单线程去切换任务(由程序员完成，而不是系统调用来做)，性能远高于多线程上下文切换，并发性更高。 3. 不需要lock机制，减少了死锁的发生。

协程的编写就是在函数中可以暂停去执行其它函数或操作，一般的函数调用是函数调用栈，但是调用完就返回了，不会保存上下文。在python中，我们可以通过生成器来暂停函数的执行。

原生异步协程

asyncio -> 异步IO库，有啥用，牛逼就完事了。开玩笑的。 作为目前最为强大的异步IO核心库之一，实现可很多功能，包括时间循环，协程调度，多线程和多进程调度。 开始之前，这下概念了解一哈： - event_loop 事件循环：程序开启一个无限的循环，程序员会把一些函数（协程）注册到事件循环上。当满足事件发生的时候，调用相应的协程函数。 - coroutine 协程：协程对象，指一个使用async关键字定义的函数，它的调用不会立即执行函数，而是会返回一个协程对象。协程对象需要注册到事件循环，由事件循环调用。 - future 对象： 代表将来执行或没有执行的任务的结果。它和task上没有本质的区别 - task 任务：一个协程对象就是一个原生可以挂起的函数，任务则是对协程进一步封装，其中包含任务的各种状态。Task 对象是 Future 的子类，它将 coroutine 和 Future 联系在一起，将 coroutine 封装成一个 Future 对象。 - async/await 关键字：python3.5 用于定义协程的关键字，async定义一个协程，await用于挂起阻塞的异步调用接口。其作用在一定程度上类似于yield。

import asyncio
import time

    await asyncio.sleep(1)
    return '消耗了{}秒时间！'.format(x)

    print("执行回调函数：",future.result())

并发http请求

我们知道我们可以使用requests来进行url请求，但是这个requests并不是异步的，而是同步阻塞，因此，不能用于异步并发请求。我们可以使用aiohttp来代替，来编程高并发HTTP服务端或客户端程序。一个简单的例子如下：

import asyncio
import time
from aiohttp import ClientSession

#多任务列表
tasks = []
url = "https://www.baidu.com/{}"

async  def get_url(url):
    async with ClientSession() as session:
        async with session.get(url) as response:
            #异步等待IO
            response = await response.read()
            print(response)
            print("Hello,World:%s"%time.time())

#多任务加入列表，2000次并发
def run():
    for i in range(2000):
        #future对象，和多线程中的future对象类似
        task = asyncio.ensure_future(get_url(url.format(i)))
        tasks.append(task)

if __name__=="__main__":
    start_time = time.time()
    loop = asyncio.get_event_loop()
    run()
    loop.run_until_complete(asyncio.wait(tasks))
    print("耗时：",time.time()-start_time)
    #耗时： 19.369337797164917
    #2000次并发只需要19s，只是在macOS下测试的。linux或者windows可能最大并发数量有限制，可以通过不同方式修改。
