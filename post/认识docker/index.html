Docker作为现在最火的容器解决方案已经风靡全球

说来惭愧，docker火了这么久，我还没有去了解，真是……，这次了解还是因为在vps上需要安装某个比较难安装的软件，看到docker一条命令就安装好，还有解决了各种依赖问题，但是就叹为观止。于是就在上网看了一下午的关于docker的知识，现在总算是有所了解了。

什么是docker

A；这软件在我电脑上运行不了啊。 B：不可能，我这运行的好好的。

嗯，这个时候你就需要docker了。

docker作为应用容器就是为了解决应用部署环境的问题，以达到一个镜像随处运行。

docker的安装方式

  • linux下安装（分ubuntu，centOS等）：参考官网
  • macOS：brew cask install docker
  • windows下安装，下载可执行文件一路向下

docker分CE和EE版本，一般个人使用免费的CE版本即可。EE是给企业用的，收钱的哟。

docker中的镜像、容器、和库

docker中的镜像和容器就向类与对象的关系，镜像是源代码，静止不变的，而容器是以镜像为基础再加上运行层的运行时，相当于程序与进程的关系。而库就是我们把docker镜像放置的地方，累似我们把代码放在Github上一样。

$ docker pull 库:[tag] #拉取或下载镜像，一个镜像有不同的版本，对应不同的tag，如果不加tag，则默认拉取last版本，即最新的版本。

$ docker run -it 镜像名 #以交互方式运行容器会进入容器内的shell环境

$ docker run -d -p 80:80 --name 容器名（任取） 镜像名 # 以后台方式运行一个并将容器的80端口映射到宿主机的80端口上。

$ docker images 或docker image ls -a #会展示所有本机内的镜像

$ docker ps -a #展示所有容器，不管是运行的还是停止的，不带-a显示运行中的容器

$ docker stop 容器ID或名字  #停止容器

$ docker start 容器ID或名字 #启动停止的容器

$ docker restart 容器ID或名字  #重启容器

$ docker rm 容器ID或名字 #删除容器，删除容器前必须先停止容器

$ docker ps -a -q #-q 选项可以只打印容器ID，我们可以搭配使用
如

$ docker rm $(docker ps -a -q) #可以删除所有容器

$ docker rmi 镜像名或ID #删除镜像

$ docker system df #查看实际镜像或容器占用的空间，这里注意我们的镜像占用不是简单的镜像相加，而很多镜像可能用到了公共的基础镜像，或者某些公共层等。

docker的镜像都是叠加层的，所以可以更加灵活和复用，不过要注意避免臃肿。

制作我们自己的镜像

在容器使用过程中，我们如果改变了容器内的内容，默认情况下，容器运行层不会保留改变的内容到镜像中，如果我们要保存在容器运行时的改变，可以使用docker commit 命令建立新的镜像。 不过我们要慎用docker commit，因为命令的执行，还有很多文件被改动或添加了。这还仅仅是最简单的操作，如果是安装软件包、编译构建，那会有大量的无关内容被添加进来，如果不小心清理，将会导致镜像极为臃肿。

此外，使用 docker commit 意味着所有对镜像的操作都是黑箱操作，生成的镜像也被称为黑箱镜像，换句话说，就是除了制作镜像的人知道执行过什么命令、怎么生成的镜像，别人根本无从得知。而且，即使是这个制作镜像的人，过一段时间后也无法记清具体在操作的。

注意： docker commit 命令除了学习之外，还有一些特殊的应用场合，比如被入侵后保存现场等。但是，不要使用 docker commit 定制镜像，定制镜像应该使用 Dockerfile 来完成。

什么是dockerfile文件

镜像的定制实际上就是定制每一层所添加的配置、文件。如果我们可以把每一层修改、安装、构建、操作的命令都写入一个脚本，用这个脚本来构建、定制镜像，那么之前提及的无法重复的问题、镜像构建透明性的问题、体积的问题就都会解决。这个脚本就是 Dockerfile。

FROM    基础镜像或/scratch #基础镜像可以是带特定服务或运行环境的如nginx、pythn等，还可以是操作系统镜像如ubuntu、centOS等。此外，还可以是一个空白镜像，所写的RUN将作为镜像的第一层，这种方式实际上很常见，对于linux下静态编译的程序来说，并不需要有操作系统提供运行时支持，所需要的一切库都已经在可执行文件里了。因此直接 FROM scratch 会让镜像体积更加小巧。使用 Go 语言 开发的应用很多会使用这种方式来制作镜像，这也是为什么有人认为 Go 是特别适合容器微服务架构的语言的原因之一。


RUN     需要执行的一些shell命令或exec格式的命令
RUN     2 #RUN的每一层都会建立一个层，这样会产生非常臃肿、非常多层的镜像。

推荐如下写法：

RUN     命令1 \
    && 命令2 \
    && 清理命令 \
    ...
#  这样就只有一层镜像加在基础镜像上，比较简洁，而且清理了一些无关的东西。
